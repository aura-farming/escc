# v1.9.0 — the digital twin: learn the rep automatically

**Type:** minor (new capability) · **Governing decision:** ADR-0019 ·
**Catalog:** 68 skills, 18 agents, 70 commands, 30 hook matchers (unchanged —
this release adds libraries and CLI verbs, not new routing surfaces)

ESCC already remembered accounts and fed a learning loop; v1.9.0 makes it
*learn the rep automatically* instead of by manual filling. The twin now
prepares the day, keeps the rep's per-account writing voice current from real
correspondence, mines reusable knowledge from every processed call, and lets
the outcomes ledger feed itself — all in-session, all behind the existing human
gates, with new rollback and observability so the rep can see and steer what it
learned. The design was code-grounded and adversarially pressure-tested before
a line was written (ADR-0019).

## What shipped

### The prepared day (accounts loop, lane L-C)

- The first `/daily` of the day runs an in-session **morning sweep**: it
  enumerates today's meeting accounts plus recently-active accounts
  (`accountMemory.listAccounts`), reconciles them against a live CRM snapshot in
  one pass (`escc reconcile` now accepts a batch `{accounts:[…]}`), refreshes
  enrichment **to a review-pack only** (never auto-applied), and stages prepared
  items the rep opens to.
- **A persistent prepared-day store** (`escc worklist add|list|done`) on the
  state-store `work_items` table. Prepared items hold **structured whitelisted
  fields only** (canonical account key, ISO meeting time, skill pointer, CRM
  as-of) — never calendar-invite text or any prospect free string — so nothing
  prospect-authored is persisted and re-injected across sessions. Session-start
  shows a counts-and-pointers "Prepared for today" line; `/daily` renders the
  full "Prepared for you" section and generates briefs live.
- Reconcile stays read-side only (writes account-memory, never CRM); the
  fail-closed send-gate still owns every send; `crm-operator` stays the sole
  CRM writer.

### The style loop keeps the rep's voice current

- `reply-handling`, `inbox-triage`, and `discovery-notes` now **build or refresh
  the per-account voice overlay** from the buyer side of the thread/transcript
  they just read (buyer text only; discovery-notes uses the quarantine-clean
  transcript-analyzer summary). Facts still come only from approved
  product-knowledge — the overlay is STYLE only (register + recurring words).
- A **downgrade guard** protects overlays: a refresh built from a thin thread no
  longer overwrites a higher-confidence overlay (the write is skipped unless
  `--force`), and every overwrite backs the prior overlay up to `<file>.bak`.
  Staleness is now actionable (`ESCC_VOICE_STALE_DAYS`).

### The knowledge loop mines itself (behind the firewall)

- `discovery-notes`, `call-review`, and `meeting-followthrough` **auto-mine
  reusable candidates** (objections, pains, competitor mentions) from the
  structured transcript summary into the operator-only candidate area — forced
  `approved:false` + `untrusted:true` (ADR-0012), quotable only after a human
  promotes them.
- Two hardening fixes: `escc product mine --from-transcript` now **refuses a
  quarantined path** (the Read-matcher hook cannot see a Bash-invoked CLI read),
  and ingestion is **capped** (`ESCC_MINE_MAX`, drops surfaced) so auto-mining
  cannot flood the review queue into a rubber stamp.

### The outcomes loop feeds itself — and can be rolled back

- `inbox-triage` **auto-attests a genuine inbound reply** as `reply_received`.
  `escc outcome record --thread <id>` **dedupes** on a fingerprint, so
  re-triaging a thread records one row, never two. Whitelisted fields only;
  prospect prose never enters the ledger.
- **`escc outcome void <id>`** rolls back a bad or fabricated outcome. Voided
  rows are filtered at the single `listOutcomes` seam, so the rollback is
  honored everywhere at once — instinct-confidence weighting, account-truth
  counts, and `outcome summary`.

### Safety and observability

- **Privacy-purge now reaches every twin-writer store** — outcomes, promises,
  work items, the notify queue, and session metrics are auto-erased on
  `--confirm`; aggregate forecast snapshots are reported for manual review. A
  content-guard test asserts **every state-store table declares a purge
  strategy**, so a future table cannot silently escape erasure.
- **`escc twin [--days N]`** — a read-only "what did the twin learn this week"
  digest: recent outcomes by type, open prepared items, knowledge candidates
  awaiting review, and pending instincts, each line pointing at the surface
  where the rep corrects or rolls it back. Counts and pointers only, never
  another approval queue.

## Deferred to a follow-up (speced, not built)

Two of the deepest slices from the design are intentionally held for a focused
follow-up, because each carries a state-store schema change and its own review
bar:

- **Fidelity instrumentation (revision-rate capture)** — deterministic
  `draft_approved_unedited` / `draft_revised` outcome types via pack-item
  content-key comparison, written to a sidecar ledger (kept off the send-gate's
  hot path), plus the dark autonomy grant/revocation model. This is the
  measurement the earned-autonomy ladder's L4 unlock depends on.
- **Machine-written resonance (the ADR-0012 amendment)** — resonance accrued as
  a descriptive, provenance-carrying count joined ONLY on the exact outbound
  content key, barred from any ranking path, applied only on explicit operator
  accept.

**L4 auto-send remains deferred to v2.0.0 behind its own ADR** — it is the first
unreviewed live outbound in the system's history, and the send-gate may only
ever tighten.

## Upgrade notes

- No breaking changes; all additions are new libraries, CLI verbs, and skill
  body steps. New env knobs: `ESCC_ACTIVE_ACCOUNT_WINDOW_DAYS`,
  `ESCC_VOICE_STALE_DAYS`, `ESCC_MINE_MAX`, `ESCC_TWIN_WINDOW_DAYS`.
- Reinstall the marketplace copy and run the live smoke test (doctor/repair →
  `/daily` → send-gate block test → one drafting flow) before relying on the
  new surfaces — a scheduled/stale copy would run older code invisibly.
