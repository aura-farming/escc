# v1.9.1 — the adversarial reviewer is enforced, and a batch actually routes

**Type:** patch (hardening + routing fix) · **Governing decision:** ADR-0020 ·
**Catalog:** 68 skills, 18 agents, 70 commands, 30 hook matchers (unchanged —
this release tightens the approval path and the routing config, not the
component catalog)

## Why

A field test surfaced the gap. Bulk-drafting ~38 emails through a hand-rolled
triage→draft→gate→create loop — in a harness where ESCC's prompt-level routing
never engaged — showed the safety layer working *exactly as designed*: the
fail-closed `pre:outbound-send-gate` blocked every draft until a per-recipient
token existed, and the four deterministic gates ran. But two things that were
supposed to happen did not:

1. **The adversarial `outbound-reviewer` never ran.** `escc outbound approve`
   minted the token on the four deterministic gates **alone** — the reviewer was
   documented as a layer "on top" but was skippable by default. Worse,
   `SECURITY.md` and `TROUBLESHOOTING.md` told operators the token *implied* a
   reviewer run. Docs claimed an enforcement the code didn't have.
2. **A batch ask never reached the blessed path.** The intent-router had no
   pattern for "mass-draft 38 emails," so the model hand-rolled the batch with
   general-purpose agents that carry none of ESCC's agent/routing contract — no
   dedicated drafter, no consolidated review-pack, no reviewer.

The send-gate held because it is a HOOK at the tool boundary. The rest didn't,
because it lived in best-effort prompt routing and an un-enforced CLI. v1.9.1
closes both — by making code match the documented intent, and only ever
*tightening*.

## What shipped

### The adversarial reviewer is required to mint a token (ADR-0020)

- `escc outbound approve` now records a per-recipient approval token only when an
  adversarial-review attestation `{verdict, confidence}` is an approval **at or
  above** the confidence floor (`ESCC_OUTBOUND_REVIEW_MIN_CONFIDENCE`, 0.8) — in
  addition to the four gates. A missing or weak review **blocks**, exactly like a
  gate failure (but writes no do-not-contact row — a thin draft is not a
  suppression signal).
- The verdict is supplied to the CLI as `--review-verdict approved
  --review-confidence 0.9 [--reviewer outbound-reviewer]`, or as
  `"review": {...}` inside `--input`. It is stamped onto the token payload, so
  `escc audit` can answer *"approved WITH what review?"*.
- **The `pre:outbound-send-gate` hook is unchanged.** The trust boundary stays
  where it was; only the *meaning* of the token is tightened at mint time. This
  is a strict tightening — a token is now harder to earn, never easier.
- **Escape hatches, both logged:** a human `--override "<reason>"` proceeds past a
  missing review (manager-signed under the strict profile, unchanged), and
  `ESCC_OUTBOUND_REQUIRE_REVIEW=off` restores the legacy four-gates-only path for
  a deliberate, supervised exception.

### A batch ask routes to the worklist on-ramp

- The intent-router `worklist` route gained high-precision batch phrasings —
  `mass/bulk (draft|email|…)`, `draft N emails`, `these N contacts`,
  `work my list/backlog/queue`, `reach out to everyone on the list` — and was
  lifted **above** the single-message routes, so "mass-draft 38 cold emails"
  surfaces the batch on-ramp instead of the single cold-email skill.
- A once-per-session **post-draft chaining-hint** (`create_draft` →
  `/escc-worklist`) nudges a nascent batch onto the sanctioned pipeline rather
  than a per-draft loop. (The `post:chaining-hints` hook matcher now includes
  `create_draft`.)
- `worklist` and `email-outbound-ops` now pass the reviewer verdict into
  `outbound approve`, and gained an explicit anti-pattern: **do not hand-roll the
  batch with general-purpose subagents** — delegate to the dedicated
  `account-researcher` / `outreach-drafter` / `outbound-reviewer` / `crm-operator`
  agents by name.

## Upgrade notes

- **This can block a flow that used to pass.** If you call `escc outbound approve`
  directly without a review verdict, it now blocks. The fix is to run the
  `outbound-reviewer` and pass its verdict (the blessed skills do this for you),
  or `--override "<reason>"`, or `ESCC_OUTBOUND_REQUIRE_REVIEW=off` for a
  supervised fallback. Gmail is still draft-only; the send-gate still fails closed.
- No component was added or removed; catalog counts and the routing char-budget
  are unchanged.

## Verification

- `npm test` green: hermetic unit suite + validators + catalog + registry pins.
  New coverage: intent-router batch routing (with negative cases proving
  single-message and prospect-list asks are untouched), the `create_draft`
  chaining-hint, and five approval-path enforcement cases (block-without-review,
  below-floor, non-approval verdict, valid-review-approves-with-audit-trail,
  kill-switch, override-bypass).
