# ESCC v1.4.0 — Drag-and-drop knowledge intake (`/ingest`)

_Released 2026-06-25 · [CHANGELOG](../../CHANGELOG.md) · [ADR-0014](../DECISIONS.md)_

Phase B of the Phase A→C roadmap set by ADR-0013. The clean, empty knowledge
layer from v1.3.0 finally has an on-ramp: drop in what you already have — sent
emails, a call transcript, a case study, a pricing or security one-pager, a
competitor doc, an ICP list — and `/ingest` routes each part to the right layer.

## Highlights

- **One guided pass to seed the knowledge layer.** `/ingest` classifies a
  dropped file, shows a dry-run routing plan, and on approval routes each part to
  a surface that already exists.
- **No new machinery, no new trust boundary.** Every leg reuses an existing
  surface; the candidate/approved firewall and the fail-closed send-gate are
  untouched.
- **Untrusted content stays quarantined.** Raw bytes are read only by a
  read-only quarantine subagent; the privileged orchestrator works from the
  cleaned structured summary.

## What's new

| Surface | Change |
| --- | --- |
| `/ingest` (`knowledge-intake` skill) | AskUserQuestion wizard: classify → dry-run → quarantine-extract → route → one review summary (mirrors `configure-escc`). |
| Routing | sent emails / brand doc → **brand-voice** (style, auto-apply); call transcript → `transcript-analyzer` → `discovery-notes` (CRM proposal + MEDDPICC) + objection/pain candidates via `escc product mine --input`; case study / pricing / security / stated claim → `escc product add` candidate; competitor doc → `battlecard` candidate + competitor-vocab suggestion; ICP / industries → `escc product vocab suggest`. |
| `content-guard-knowledge-intake` | Test-side proof of the `/ingest` invariants. |

## How to use

```text
/ingest
# then describe or attach the file(s); review the dry-run routing plan;
# approve to route. One review summary lands at the end.
```

## The guarantee

**Only STYLE (voice) and account CONTEXT auto-apply.** Every product **CLAIM**
enters as an operator-reviewed candidate (`approved:false` + `untrusted:true`,
forced by `appendCandidate`) and becomes quotable only when a human runs
`escc product approve`. `/ingest` **never** uses `escc product mine
--from-transcript` (which reads raw bytes in the CLI and bypasses the
quarantine hook) — it extracts via a read-only subagent and ingests the
structured result with `--input`.

## Upgrade notes

- **No migration.** `/ingest` is installed with the cross-cutting skills; its
  `skills-foundation` dependency guarantees `product-knowledge` is present to
  route into.
- **Catalog:** 66 skills, 68 commands (CI-pinned); command registry regenerated.

## Scope & deferred

- account-memory has no append CLI (`appendEvent` is Node-only / hook-populated),
  so `/ingest` routes account context through `discovery-notes` rather than
  writing memory directly.
- A file at a quarantine path or an `.eml` / `.msg` / `.mbox` cannot be read by
  the read-only subagent (nothing sets `ESCC_QUARANTINE_CONTEXT`), so the skill
  asks the user to paste or re-save such content as plain text. Hardening that
  gap (a per-subagent quarantine context) is **deferred**.
